# Intellect Forge Audit — 2026-04-16

## Summary

The Intellect Forge codebase demonstrates **strong sourcing discipline** on implementation mechanics, with most training parameters correctly grounded in canonical research (Jaeggi 2008, Carpenter et al. 1990, Engle Lab, Levitt 1971, Watson & Pelli 1983). However, three moderate-to-high severity issues were identified:

1. **Assessment module misleadingly labels procedural matrix items as "ICAR"** — undermines claim of validated assessment
2. **Compound EF task response mapping is arbitrary** — not grounded in any research task, though internally consistent
3. **Research dossiers are AI-generated and drive critical design decisions** — citations appear real but documents' synthesis role needs transparency

Overall trust level: **Medium-High on task mechanics, Medium on assessment validity.**

---

## Findings

### 🔴 Needs action

#### Issue 1: Misleading "ICAR" labeling in transfer assessment

**Location:** `/c/dev/brain-train/src/core/modules/assessment/assessment-module.ts` line 177
```typescript
taskId: 'icar-matrix',  // ← ICAR label but using procedural generator, not licensed items
```

**What's wrong:**
- The assessment module saves transfer-assessment results with `taskId: 'icar-matrix'`
- The matrix items are **procedurally generated** via `generateMatrix()` from `relational-module`
- **ICAR** (International Cognitive Ability Resource, icar-project.com) is a specific public-domain assessment with licensed items
- Users / researchers / clinicians reading "ICAR" will expect copyrighted, validated ICAR items, not algorithmic Carpenter-rule matrices
- This is a validation claim that is false

**Why it matters:**
- If a user compares Intellect Forge "ICAR matrix" scores to published ICAR norms, the comparison is invalid
- Research papers citing the app will incorrectly assert ICAR-validated transfer assessment
- Misrepresents training validity

**Suggested fix:**
- Rename `taskId` to `'matrix-reasoning'` or `'procedural-matrix'`
- Document clearly in the app UI that this is "Matrix Reasoning (procedurally generated, not ICAR items)"
- If future work ingests real ICAR items from icar-project.com, create a separate task ID

**Severity:** HIGH — direct misrepresentation of assessment type

#### Issue 2: Compound EF response mapping is arbitrary (no research grounding)

**Location:** `/c/dev/brain-train/src/core/modules/compound-ef/ef-module.ts` lines 12–17
```typescript
const RESPONSE_MAP: Record<string, { color: string; shape: string; size: string }> = {
  a: { color: 'red',    shape: 'circle',   size: 'small' },
  s: { color: 'blue',   shape: 'square',   size: 'small' },
  k: { color: 'green',  shape: 'triangle', size: 'large' },
  l: { color: 'yellow', shape: 'diamond',  size: 'large' }
};
```

**What's wrong:**
- The response map couples **all three dimensions** (color, shape, size) per key
- This is not any canonical EF task — not Stroop, not Flanker, not Wisconsin Card Sort, not Stop-Signal, not Task-Switching per se
- The specific pairings (A=red+circle+small, etc.) appear arbitrary
- Research/03-compound-executive.md (lines 109–120) documents how real compound tasks are constructed — none use this "all dimensions at once" mapping

**Why it matters:**
- Users training on this task cannot easily compare their performance to published norms for any standard EF test
- If the mapping is arbitrary, changing it in a future iteration could inadvertently alter task difficulty or construct validity
- Makes it hard to validate near-transfer to real EF tasks

**Suggested fix:**
- **Option A (MVP acceptable):** Document in the UI/research that this is "Custom Multi-Dimensional Rule Task" — no specific task, bespoke design. Keep the mapping unchanged and fixed.
- **Option B (research-grounded):** Either (a) base the rule-mapping on a published study's stimulus set, or (b) clearly state it's a novel synthetic task designed to interleave flanker-like interference + task-switching + stop-signal, with the mapping chosen for ergonomic reasons (e.g., "A and S require left-side/small/cool-side choices, K and L require right/large/warm-side choices for motor balance").

**Severity:** MEDIUM — confusing but not false; task is internally consistent. Documenting as "bespoke design" resolves the issue.

#### Issue 3: Research dossiers are AI-generated and drive core design decisions

**Locations:** `/c/dev/brain-train/research/` (all 14 .md files marked as AI-generated in README or commit history)

**What's wrong:**
- All research dossiers were authored by an LLM (per background brief)
- These dossiers are load-bearing: they inform the Carpenter rule definitions, UFOV parameters, QUEST thresholds, n-back d-prime promotion rules, calibration methodology, orchestrator phase boundaries, etc.
- No dossier underwent expert review; citations were spot-checked during development but not exhaustively verified
- Spot checks found:
  - Valid author citations (Jaeggi, Carpenter, Levitt, Watson & Pelli, Cepeda, etc.) with correct years and journals
  - Valid paper URLs (PNAS, Nature, Psychonomic Bulletin & Review all resolve)
  - Some citations appear correct in abstract but the AI synthesis of their key claims was not independently verified (e.g., "Bollen 2019 is a null result" — this claim is stated in the research file but I did not independently verify that the paper actually reports null transfer)

**Why it matters:**
- If a dossier synthesized a paper's findings incorrectly (e.g., misreported effect sizes, conflated studies), the implementation would propagate that error
- Dossier-sourced numbers (e.g., "Jaeggi's target is 79.4% correct convergence on 3-down/1-up" per research/10-adaptive-algorithms-deep.md) are foundational to task difficulty
- Users trusting the app are indirectly relying on unreviewed secondary sources

**What's verified as sound:**
- N-back consonant set (C,H,K,L,Q,R,S,T) from Jaeggi 2008 ✅
- N-back timing (500 ms stim, 2500 ms ISI) from Jaeggi 2008 ✅
- d-prime ≥ 1.5 promotion threshold — research/02-working-memory.md line 205 cites this as "maximum N sustained at d′ ≥ 1.5" and it matches Jaeggi protocol ✅
- SSD staircase step 50 ms per Verbruggen consensus — research/03-compound-executive.md line 68 matches Verbruggen et al. 2019 ✅
- Brier scoring formula — matches research/05-calibration.md line 13 and is textbook correct ✅
- Half-range 50–99% confidence slider — research/05-calibration.md line 64 cites CFAR and notes this is canonical ✅

**Suggested fix:**
- **Immediate:** Add a banner to the research/ directory and to the README stating: "Research dossiers are AI-synthesized from primary literature. Implementation parameters have been spot-checked against source papers; however, these syntheses have not undergone formal expert review. Users relying on this content for publication should verify claims independently."
- **Medium-term:** Prioritize expert review of: (a) QUEST default parameters (research/10-adaptive-algorithms-deep.md §2), (b) compound EF construction rules (research/03-compound-executive.md §1–2), (c) any papers cited for effect sizes or meta-analyses driving protocol (orchestrator intervals, dosing).
- **Do NOT:** wait for exhaustive review to ship; spot checks show the dossiers are broadly sound.

**Severity:** MEDIUM — existing parameters appear correct, but the synthesis layer needs transparency.
